# CloudDuka Architecture

## 1) High-level system view
CloudDuka is a full-stack commerce platform with:
- **Frontend**: React single-page app (`frontend/`) for marketplace browsing, cart, checkout, and owner workflows.
- **Backend**: FastAPI service (`backend/server.py`) exposing auth, product, cart, orders, payments, customer checkout, analytics-friendly endpoints, and shop/subscription APIs.
- **Data layer**: MongoDB collections accessed via Motor/PyMongo abstractions in the backend.

The frontend calls backend REST endpoints over HTTP. Authentication uses bearer JWTs generated by the backend.

## 2) Frontend architecture

### Core structure
- `src/App.js`: application route composition and page registration.
- `src/pages/`: screen-level views such as `HomePage`, `CartPage`, `CheckoutPage`, `OrderSuccess`.
- `src/components/`: reusable UI pieces (marketplace cards, header, bottom nav, confirm modal, design-system controls).
- `src/hooks/`: business state and async orchestration hooks.
- `src/lib/`: small utilities such as analytics/event tracking.

### State and data flow
- `useCart` (Zustand) is the central client-side cart state manager.
- `useCheckout` coordinates checkout submission, idempotency handling and client-side guard rails.
- `useHomeData` fetches homepage payloads with cancellation-safe behavior.

### UX resilience patterns
- Defensive rendering for incomplete product/store payloads.
- Loading/disabled states for async operations.
- Explicit route pages for cart/checkout/success, allowing deep-linking and mobile-friendly flows.

## 3) Backend architecture

### API layer
`backend/server.py` contains FastAPI route handlers and helper functions for:
- Auth and role resolution.
- Shop and subscription checks.
- Product/catalog operations.
- Owner cart/checkout/payment endpoints.
- Customer cart/checkout/order history endpoints.

### Security and access
- JWT bearer auth resolves user context and role (`owner`, `shopkeeper`, `customer`).
- Shop access validation protects cross-shop data leakage.

### Subscription handling
- `normalize_shop_subscription` normalizes plan/status from shop data.
- `check_shop_subscription` enforces active status and `online`-feature checks.
- Backward compatibility allows legacy shops lacking embedded `subscription` to avoid accidental hard-blocking while preserving explicit-plan enforcement.

## 4) Checkout and order lifecycle

### Customer flow
1. Customer authenticates.
2. Adds items to customer cart.
3. Submits checkout with payment method.
4. Backend validates cart/shop/product state, creates order + order items + payment records.
5. Customer cart is cleared and order is returned for success UI.

### Owner/POS flow
1. Staff/owner adds items to POS cart.
2. Checkout creates order and payment state.
3. Payment transitions update order/payment status.

## 5) Reliability and test strategy
- Backend test suite (`pytest`) validates auth, cart, checkout, payments, subscription gating, and customer order dashboards.
- Frontend build (`npm run build`) validates compile-time integrity.
- Frontend test command (`npm test -- --watchAll=false --passWithNoTests`) validates runner + existing test inventory.

## 6) Operational notes
- In locked-down environments with restrictive proxy variables, run npm install with proxy env vars unset (see root `README.md` troubleshooting).
- Keep backend and frontend validation in PR checks: dependency install, backend tests, frontend test runner, and production build.

## 7) Functional map by actor (owner, shops, online store, customer)

### Owner (merchant account)
- Registers/logs in and manages the primary shop profile.
- Maintains product catalog (create/update/activate/deactivate items, stock, pricing).
- Operates POS-style cart and checkout flows for in-person orders.
- Manages payments and payment confirmations by channel (cash/mobile/web where configured).
- Views/manages orders and order lifecycle status transitions.
- Manages subscription configuration (plan/status) and shop visibility for online storefront exposure.

### Shopkeeper/staff
- Uses authorized shop context assigned by owner.
- Performs day-to-day operational tasks: cart handling, order processing, stock-aware checkout.
- Cannot access shops outside assigned membership scope.

### Shop / online storefront
- Public store discovery endpoints expose only eligible online-capable shops/products.
- Store pages surface catalog, category grouping, and stock-sensitive product availability.
- Online storefront checkout enforces shop-level feature gating and product validity checks.

### Customer
- Registers/logs in as customer role with isolated access model.
- Browses public stores/products and builds a customer cart.
- Updates/removes cart items and proceeds through checkout flow.
- Receives order success state and can view personal customer order history/details.
- Cannot access or view other users’ orders due per-user order ownership checks.

### Cross-cutting flows
- **Authentication/Authorization**: JWT bearer token; role-driven endpoint protection.
- **Cart domains**: owner/staff POS cart and customer cart are separate logical domains.
- **Checkout safety**: validates product active status, stock, single-shop constraints, and idempotency keys for customer checkout.
- **Observability/analytics**: frontend emits lightweight interaction events (e.g., add-to-cart, checkout started/success).
